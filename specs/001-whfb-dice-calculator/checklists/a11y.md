# Accessibility Requirements Quality Checklist

**Purpose**: Validate that accessibility requirements (keyboard reach, ARIA labelling, focus, contrast, reduced motion, screen-reader behaviour) are written with enough rigor that the implementer doesn't have to invent policy.
**Created**: 2026-05-06
**Feature**: [spec.md](../spec.md)
**Audience**: Author self-check before `/speckit-implement`. Each item asks whether the spec / plan / research _writes down_ the rule — not whether the implementation passes axe.

## Requirement Completeness

- [ ] CHK001 Are keyboard navigation requirements specified for every interactive control (NumberStepper +/-, SaveSelect, ModifierToggleRow, top-bar buttons, Tooltip)? [Completeness, Spec §FR-035]
- [ ] CHK002 Are ARIA-label requirements pinned for the icon-only top-bar buttons (Settings gear, Reset, Share, Load Preset trigger)? [Completeness, Spec §FR-003 / FR-035]
- [ ] CHK003 Are ARIA-label requirements pinned for each ModifierToggleRow (label + numeric effect, e.g. "Charging, +1 to hit")? [Completeness, Research §4]
- [ ] CHK004 Is the NumberStepper value-change announcement (`aria-live="polite"` or equivalent) specified so screen readers narrate stat changes? [Completeness, Research §4]
- [ ] CHK005 Are focus-visible style requirements specified, distinguishing keyboard focus from mouse focus? [Gap]
- [ ] CHK006 Is the focus order across the page explicitly defined (top bar L→R → card 1 inputs → card 1 modifiers → card 1 result-skip → card 2 → ...)? [Gap]
- [ ] CHK007 Are dismissal requirements specified for the Tooltip (Esc dismisses, focus restored)? [Completeness, Research §4]
- [ ] CHK008 Are dismissal requirements specified for the SaveSelect dropdown (Esc closes, focus returns to trigger)? [Gap]
- [ ] CHK009 Is the empty Load Preset Select's accessibility state pinned (does it announce "no presets" to screen readers, or is it disabled)? [Gap, Spec §FR-029]

## Requirement Clarity

- [ ] CHK010 Is "modifier active state conveyed by more than colour alone" (FR-036) pinned to concrete cues (switch thumb position, label position, badge value)? [Clarity, Spec §FR-036]
- [ ] CHK011 Are minimum contrast ratios specified for the dark theme (e.g., WCAG AA 4.5:1 for body text, 3:1 for large text and UI components)? [Gap]
- [ ] CHK012 Is the focus-ring color and width specified to remain visible against the dark card background and the dark page background? [Gap]
- [ ] CHK013 Is the gold-orange result number's contrast against the card background pinned to a specific ratio (it's the most prominent text on the card)? [Gap, Spec §FR-010]
- [ ] CHK014 Is the green-active modifier accent's contrast against the inactive state pinned (so the toggle's on/off difference is reliably distinguishable)? [Gap, Spec §FR-009]

## Requirement Consistency

- [ ] CHK015 Are ARIA-labelling patterns consistent across NumberStepper / SaveSelect / ModifierToggleRow / TopBar buttons (same shape: visible label + invisible context if needed)? [Consistency]
- [ ] CHK016 Are keyboard shortcut requirements consistently absent (or, if any, consistently documented across all four cards)? [Consistency, Gap]
- [ ] CHK017 Are landmark / heading-level requirements consistent (e.g., page H1, card H2, section H3)? [Gap]

## Coverage

- [ ] CHK018 Are reduced-motion requirements defined for any state transitions (toggle thumb, modifier badges)? [Gap]
- [ ] CHK019 Are color-blind verification requirements pinned (e.g., "test under deuteranopia / protanopia simulation in Storybook a11y addon")? [Gap, Spec §SC-006]
- [ ] CHK020 Are Storybook a11y addon (axe) violations gated as CI failures, or only surfaced as warnings? [Gap, Tasks T043]
- [ ] CHK021 Is there a mandate to use Radix primitives (which are keyboard-correct by default) for every interactive element rather than custom implementations? [Coverage, Research §4]
- [ ] CHK022 Are screen-reader announcements specified for the result panel transitions (when a modifier toggles and the required roll changes)? [Gap]
- [ ] CHK023 Are the "—" / "No save" / "Auto-fail" boundary-state strings specified to be read aloud by screen readers (not just rendered as glyphs)? [Coverage, Spec §FR-T05]

## Measurability

- [ ] CHK024 Can SC-005 ("100% of interactive controls reachable by keyboard") be objectively verified by a Playwright keyboard-walk test? [Measurability, Spec §SC-005]
- [ ] CHK025 Can SC-006 ("color-blind users can identify modifier state from non-colour cues") be tested without subjective judgment (e.g., by asserting both `aria-checked` and a non-color visual cue)? [Measurability, Spec §SC-006]
- [ ] CHK026 Are concrete acceptance thresholds pinned for axe violations (zero serious / zero critical, or a specific allowed-list)? [Gap]

## Notes

- Multiple `[Gap]` markers here are expected — the spec explicitly opts in to "baseline accessibility" and leaves contrast minima, focus-ring styling, and reduced-motion to the implementer.
- Items are ordered roughly from "spec-pinned" → "should be pinned before implementation" → "implementer's-call but worth flagging".
